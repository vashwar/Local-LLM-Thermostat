## Problem Statement

The user owns a basic (non-learning) Nest thermostat and spends ~5 minutes daily manually adjusting it. They live in a hot climate and want energy-efficient, hands-off climate control — but can't justify $250+ for a Nest Learning Thermostat. They want the learning and optimization features for free, running locally on their PC, with zero data leaving their home.

## What Makes This Cool

An actual AI agent — not a rule engine, not a cron script — that **reasons** about your home. Your wife texts "we're heading to dinner," and the agent thinks: "They'll be gone 2 hours, it's 98F outside, I should setback now and pre-cool 30 minutes before they return." That's not automation. That's intelligence.

The "whoa" moment: friends watch the Telegram chat and see the agent explain its reasoning in plain English before adjusting the temperature.

## Constraints

- **Budget:** Near-zero. No recurring costs. One-time $5 Google SDM Device Access registration fee accepted.
- **Privacy:** All data stays local. No cloud AI services. Local LLM only (llama.cpp server).
- **Hardware:** Runs on the user's Windows 11 PC (needs ~4-8GB RAM for Qwen 4B model).
- **Platform:** Windows 11. Agent runs as a background process. All file paths use `pathlib` for portability.
- **Interface:** Telegram bot only. No dashboard, no web UI.
- **Users:** The user + their wife. Both control via Telegram. Bot restricted to whitelisted chat IDs only.
- **Safety:** Hard-coded guardrails prevent equipment strain. Manual overrides always win.

## Premises

1. The existing non-learning Nest can be controlled programmatically via Google's Smart Device Management (SDM) API.
2. A local LLM (Qwen 4B via llama.cpp server) can make good temperature decisions given structured context. 4B is smaller — tool-calling reliability needs validation, but the focused domain (thermostat only) keeps the reasoning simple. Mitigation strategy defined below.
3. A 20-minute evaluation loop is the right cadence — not too aggressive for HVAC, not too lazy for comfort.
4. The user's wife will use the Telegram bot — but the system must also work without human input.
5. Weather API + indoor temp + user messages are sufficient inputs to start. No occupancy sensors needed.

## Approaches Considered

### Approach A: Home Assistant + LLM Brain

Use Home Assistant as the infrastructure layer (Nest control, Telegram bot, weather). Bolt on a local LLM via custom component for the reasoning layer.

- **Effort:** M (human: ~2 weeks / CC: ~2 hours)
- **Risk:** Low
- **Pros:** Most infrastructure pre-built, battle-tested, easy to extend
- **Cons:** Heavy dependency, requires learning HA, less satisfying to build

### Approach B: Pure Python Script

Simple Python app with a 20-minute loop: read sensors + weather, call local LLM, set thermostat. Telegram bot runs alongside.

- **Effort:** S (human: ~1 week / CC: ~45 min)
- **Risk:** Low
- **Pros:** Simplest, no heavy dependencies, ships fastest
- **Cons:** More boilerplate, less agent-like, basic rule generation

### Approach C: LLM Decision Agent (CHOSEN)

A proper AI agent running locally (llama.cpp server + Qwen 4B) that receives structured context and returns decisions as JSON. The Agent Core pre-fetches all data and executes the LLM's decisions. Telegram is both the control interface and how the agent communicates.

- **Effort:** M (human: ~2 weeks / CC: ~1 hour). The CC effort delta over Approach B is ~15 minutes; the additional human effort (~1 week of testing, prompt tuning, and integration) is the real investment.
- **Risk:** Medium
- **Pros:** Truest to the vision, best show-off factor, handles novel situations, fast inference via llama.cpp
- **Cons:** 4B model may need careful prompt engineering for reliable tool-calling

## Recommended Approach

**Approach C: LLM Decision Agent** — because it's the version that produces the "whoa" moment. C delivers the actual autonomous agent the user envisioned. The additional human effort over B (~1 week of prompt tuning and integration testing) is worth it for the qualitative leap in capability.

### Architecture

|   Telegram Bot   |<--->|   Agent Core     |<--->| llama.cpp server |

+------------------+     +--------+---------+     +------------------+

             |  Nest   |  | Weather |  | Schedule |

### Components

1. **Agent Core (Python)** — The orchestrator. Two triggers for LLM evaluation:
   - **Scheduled loop (every 20 minutes):** Assembles full context (indoor temp, outdoor weather, schedule, recent user messages) and sends it to the LLM. This is the autonomous backbone.
   - **User-triggered (on Telegram message):** When a user sends a message, the Agent Core immediately runs an evaluation cycle with the message included in context. This provides near-real-time conversational responsiveness. The 20-minute timer resets after a user-triggered cycle to avoid redundant evaluations.

   For multi-turn exchanges (e.g., "We're heading out" → "When should I expect you back?" → "Around 6pm"), each user message triggers a new LLM call with the last 10 messages included in the `recent_messages` field. The LLM sees the conversation history and responds contextually.

   Parses and validates the LLM's structured JSON response. Executes actions. Enforces guardrails. Logs all decisions to SQLite.

2. **LLM (llama.cpp server)** — Qwen 4B model, already available on the user's PC. llama.cpp server provides an OpenAI-compatible API endpoint (`/v1/chat/completions`) with faster inference than Ollama. Receives structured context and returns decisions in a fixed JSON schema (see Prompt Engineering below).

3. **Telegram Bot (python-telegram-bot)** — Receives natural language messages from whitelisted chat IDs only (user + wife). Rejects/ignores all other senders. Passes messages to the Agent Core as context. Also receives agent status messages. Supports `/history` command to retrieve recent decision logs.

4. **Nest API Client (Google SDM API)** — Reads current temperature, humidity, HVAC mode. Sets target temperature. Handles OAuth 2.0 token refresh automatically. Respects rate limits.

5. **Weather Client (OpenWeatherMap free tier)** — Fetches current conditions + 48-hour forecast. Free tier: 1000 calls/day (more than enough for 20-min loops). Caches last-known forecast for use during API outages.

6. **Schedule Store (SQLite)** — Stores user-provided schedule (e.g., "work 9-5 weekdays"), recent user messages, and decision history. Updated by user Telegram messages.

### Prompt Engineering

The LLM does NOT use native function calling. Instead, it receives a structured prompt and returns a fixed JSON schema. This is more reliable for a 4B model than tool-calling.

**System prompt template:**
```
You are a smart thermostat agent. Your job is to decide the optimal
temperature setting for a home based on the context provided.

CURRENT STATE:
- Indoor: {temp}F, {humidity}% humidity
- Outdoor: {current_temp}F, forecast: {forecast_summary}
- HVAC mode: {mode}, current target: {target}F
- Schedule: {schedule_context}
- Recent messages: {recent_messages}
- Time: {current_time}, Day: {day_of_week}

Respond ONLY with valid JSON in this exact format:
{
  "action": "set_temperature" | "no_change",
  "temperature": <number between 60-85, or null if no_change>,
  "reasoning": "<1-2 sentence explanation>",
  "message_to_user": "<optional message to send via Telegram, or null>"
}

NOTE: "action" determines the thermostat action only. "message_to_user"
is independent — the agent can send a Telegram message alongside ANY
action. For example, action="set_temperature" + message_to_user="Pre-
cooling for the afternoon heat" is a temperature change WITH a
notification. action="no_change" + message_to_user="All good, holding
at 74F" is a status update with no thermostat change.
```

**Validation layer in Agent Core:**
1. Parse LLM response as JSON. If malformed, retry once with "Respond ONLY with valid JSON" appended.
2. Validate `temperature` is within 60-85 range (guardrail enforcement).
3. Validate `action` is one of the allowed values.
4. If validation fails after retry, log the error and take no action (failsafe).

**Fallback if Qwen 4B proves unreliable:** If structured JSON output fails >20% of the time after prompt tuning, upgrade to Qwen 7B or Mistral 7B. The llama.cpp server swap is a config change, not a code change.

### Tools the LLM Has Access To

The LLM doesn't call tools directly — the Agent Core pre-fetches all data and includes it in the prompt context. The LLM's output (JSON) determines what action the Agent Core takes.

| Data Source | Description | Fetched By |
|-------------|-------------|------------|
| Indoor conditions | Current temperature + humidity from Nest | Agent Core (pre-fetched every cycle) |
| Weather | Current + 48h forecast outdoor conditions | Agent Core (pre-fetched every cycle) |
| Schedule | User-provided schedule + recent Telegram messages | Agent Core (read from SQLite) |
| HVAC runtime | Estimated runtime by tracking heating/cooling mode duration between cycles | Agent Core (calculated from state history in SQLite) |

| Action | Description | Executed By |
|--------|-------------|-------------|
| `set_temperature` | Set target temperature on Nest | Agent Core (after guardrail validation) |
| `send_message` | Send a Telegram message to the user | Agent Core (from `message_to_user` field) |
| `no_change` | Maintain current temperature | Agent Core (log only) |

### Safety Guardrails (Hard-Coded, Non-Negotiable)

These are NOT controlled by the LLM. They're enforced in the Agent Core before any action executes:

- **Temperature bounds:** Never set below 60F or above 85F, regardless of what the LLM decides.
- **Rate limiting:** No more than 2 temperature changes per hour (prevents HVAC short-cycling).
- **Manual override priority:** If the user manually adjusts the Nest (physically or via Nest app), the agent continues its evaluation loop but suppresses `set_temperature` actions for 2 hours. After the 2-hour backoff, the agent resumes control starting from the current Nest temperature. During backoff, incoming Telegram messages are still passed to the LLM for evaluation. If the LLM's response includes a `set_temperature` action, the backoff is cancelled and the action executes. If the LLM returns `no_change`, the backoff continues.
- **Failsafe:** If the LLM fails to respond, returns malformed JSON, or fails validation after retry, maintain current temperature. Never change temp on error.
- **HVAC protection:** Minimum 5-minute gap between HVAC mode transitions (e.g., switching from cooling to heating or vice versa). This does not apply to target temperature changes within the same mode.

### Failure Modes

Each external dependency can fail. The agent degrades gracefully:

| Dependency | Failure | Behavior |
|------------|---------|----------|
| **Nest SDM API** | Unreachable / auth error | Log error. Notify user via Telegram if possible. Hold current state. Retry next cycle. |
| **Weather API** | Rate limited / down | Use last-known cached forecast. If cache is >6 hours old, degrade to conservative setpoints (no pre-cooling, maintain current target). |
| **Telegram** | Bot disconnected | Continue autonomous evaluation loop. Queue status messages. Deliver queued messages when reconnected. |
| **llama.cpp server** | Crashed / unresponsive | Maintain current temperature (failsafe). Log error. Notify via Telegram when reconnected. |
| **SQLite** | Corrupted / locked | Agent operates in stateless mode (no schedule context, no history). Log error. |

### Telegram Authentication

The Telegram bot restricts access via **whitelisted chat IDs**:

- On first setup, the user configures allowed chat IDs in `config.yaml`.
- Every incoming message is checked against the whitelist before processing.
- Messages from unauthorized senders are silently ignored (no response, no error — to avoid leaking the bot's existence).
- The `/status` command returns current state (indoor temp, target, HVAC mode, last decision).
- The `/history` command returns the last 10 decisions with timestamps and reasoning.

### Logging & Observability

All LLM interactions and decisions are logged to SQLite:

| Table | Fields | Retention |
|-------|--------|-----------|
| `decisions` | timestamp, indoor_temp, outdoor_temp, target_temp, action, reasoning, llm_raw_response | 90 days |
| `messages` | timestamp, sender_chat_id, message_text, agent_response | 90 days |
| `errors` | timestamp, component, error_type, details | 90 days |
| `hvac_state` | timestamp, mode, target_temp, is_running | 90 days (used for runtime estimation) |

Log rotation: a daily cleanup task deletes records older than 90 days.

### Startup & Recovery

- **Windows service:** Agent runs as a background process managed by NSSM (Non-Sucking Service Manager) or Windows Task Scheduler with "restart on failure" enabled.
- **Startup sequence:** On launch, the agent reads last known state from SQLite (last target temp, last HVAC mode, last evaluation time). If the gap since last evaluation is >40 minutes (2 missed cycles), it immediately runs an evaluation cycle. Otherwise, it waits for the next scheduled cycle.
- **Crash recovery:** All state is persisted to SQLite on every cycle. A mid-loop crash loses at most one cycle's worth of data. The agent resumes cleanly on restart.
- **llama.cpp server:** On startup, the Agent Core checks if llama.cpp is already listening on the configured port (HTTP health check to `/v1/models`). If so, reuse it. If not, start it as a subprocess. This prevents orphaned processes on crash/restart from consuming double RAM. Alternatively, llama.cpp can be managed as a separate NSSM service with independent lifecycle.

## Future Enhancements (Post-MVP)

These are deferred to keep the MVP focused:

- **Rule generation with confidence scores** — Agent observes patterns ("user always texts 'heading home' around 5:30pm") and proposes rules. High-confidence rules execute automatically. Low-confidence rules ask for confirmation. Requires pattern detection and confidence calibration.
- **Seasonal adaptation** — Agent notices cooling runtime increasing in June, suggests adjusting setback temperatures.
- **Energy monitoring** — If the SDM API exposes energy telemetry for the user's Nest model, add `get_energy_log` tool for runtime tracking and savings reports.
- **Multi-zone support** — If the user adds more thermostats later.

## Open Questions

1. **Which Nest model exactly?** The SDM API support varies by generation. Need to confirm the specific model to verify API capabilities.
2. **PC always on?** The system needs the PC running 24/7 (or at least during active hours). Is the user okay with that, or should we add a sleep/wake schedule?
3. **Qwen 4B structured output reliability** — Need to test whether Qwen 4B can reliably produce the fixed JSON schema via llama.cpp server. Mitigation: validation + retry in Agent Core, fallback to larger model if >20% failure rate.
4. **Google SDM API setup complexity** — Requires multiple steps (see Next Steps). Has the highest setup risk and should be done first.

## Success Criteria

- Agent runs autonomously for 24+ hours without intervention
- Wife can text it naturally and get sensible responses
- HVAC runtime comparison: compare daily runtime hours for the first week of agent control vs. a logged baseline week of manual control (even 10% reduction is a win)
- At least one friend says "whoa" when shown the Telegram chat
- Agent explains its reasoning in plain English before every temperature change
